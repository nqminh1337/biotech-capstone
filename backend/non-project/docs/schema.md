## tables and explanation

### users

```sql
CREATE TABLE users (
  user_id INT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  first_name VARCHAR(100) NOT NULL,
  last_name VARCHAR(100) NOT NULL,
  email VARCHAR(100) NOT NULL UNIQUE,
  status CHAR(8) NOT NULL, -- online vs offline?
  track_id INT NOT NULL,
  state_id INT -- or should this be CHAR?
  FOREIGN KEY (state_id) REFERENCES country_states(state_id)
)
```

some questions:

1. can we change first and last name to varchar instead of char?
2. what is stored in status? is it just online/offline? or activated/deactivated? in this case can we change to BOOL (activated)?
3. why is track_id a char and state_id an int? shouldn't they both be int? or should state be a char? e.g. NSW, QLD. in this case, is it only required if track_id maps to "Australia"?
4. do we need BIGINT for user_id? and this follows for other IDs. INT can store like 2 billion values

some questions:

1. with the way this is set up, every year when profiles "refresh" this means we'll have to clear the profile tables and reinsert users. e.g. we have to somehow manually iterate through our users table, then insert their ID into the respective profile table to associate them with a specific profile. maybe something to consider, it sounds a bit cumbersome. maybe each user has a profile role, instead of having separate profile tables.
2. in the student_profile table, what is parent_guardian_flag? is it a bool to indicate if they have a parent/guardian? if so, can we change to BOOL?
3. in the mentor_profile table, it might be worth having a separate table for the wwcc (Working With Children Check) details. we may have to note expiration dates, state, and other relevant information. different states are important because theres differing format for these certificates - if we want to pull details and validate them, we'd need to parse through their certificate, store this in a separate table, and perhaps have a link to an external file store e.g. Blob. **TLDR**: consider a separate table for WWCC details.
4. for fixed values such as interest or year level, maybe consider using the ENUM data type e.g.

```sql
CREATE TYPE enum_name AS ENUM ('value1', 'value2', 'value3');
```

5. can the background table just be an ENUM?? or do we need to sort by backgrounds? if the latter then should be okay to have a separate table.

### sessions

```sql
CREATE TABLE sessions (
  session_id INT PRIMARY KEY,
  user_id INT NOT NULL,
  access_datetime TIMESTAMP NOT NULL,
  isLoggedin BOOLEAN NOT NULL,
  FOREIGN KEY (user_id) REFERENCES users(user_id)
)

CREATE TABLE alerts (
  alert_id INT PRIMARY KEY,
  session_id INT NOT NULL,
  error_reason VARCHAR(255) NOT NULL,
  alert_timestamp TIMESTAMP NOT NULL,
  FOREIGN KEY (session_id) REFERENCES sessions(session_id)
)
```

some questions:

1. looks like `new_column` in the diagram is a typo?
2. should we add a `created_at` timestamp to the alerts table to track when the alert was generated?
3. is there a need for a `resolved` boolean flag in the alerts table to indicate if an alert has been addressed?
4. im not sure if `isLoggedin` is the best way to track sessions - it might fall out of sync. might be better to derive if a user is active based on timestamps - a session can be active if theres no `ended_at` or `expired_at` timestamp.
5. just a bigger one on session architecture, i think we should consider how sessions are created, managed, and terminated. e.g. sessions should expire automatically, have an idle timeout, they should be revocable as well. maybe we would need a `last_activity_at` timestamp which we update on each authenticated request, an `expires_at` timestamp to indicate when the session should be considered expired based on TTL e.g. 30mins or 2hrs etc. we could have a `ended_at` for logout purposes or `revoked_at` for admin control of sessions. then we can derive `is_active` by checking that ended_at, revoked_at, expired_at is null.

### groups and group members

```sql

CREATE TABLE groups (
  group_id INT PRIMARY KEY,
  group_name VARCHAR(200) NOT NULL,
  track_id INT NOT NULL,
  created_at TIMESTAMP DEFAULT NOW(),
  deleted_flag BOOLEAN NOT NULL,
  deleted_at TIMESTAMP
);

CREATE TABLE group_members (
  group_member_id INT PRIMARY KEY,
  group_id INT NOT NULL,
  FOREIGN KEY (group_id) REFERENCES groups(group_id),
  FOREIGN KEY (group_member_id) REFERENCES users(user_id)
);
```

some questions:

- this looks solid honestly no suggestions i can think of
- we should make group_member_id and group_id a composite key so that there are no duplicates. this still allows users to be part of multiple groups, but no group has the same user twice
